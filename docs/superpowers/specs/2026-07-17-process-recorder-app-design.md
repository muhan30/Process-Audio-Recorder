# 进程录音机 APP 设计文档

- 日期：2026-07-17
- 状态：已获用户批准（头脑风暴三轮确认）
- 上游文档：[PROJECT_WIKI.md](../../../PROJECT_WIKI.md)（现有代码结构与已知问题 K1–K8）

## 1. 背景与目标

将现有命令行工具 ProcessAudioRecorder 发展为一款**自用**的进程隔离录音 APP。

**核心场景**：用户与朋友微信通话时，录下双方完整通话（对方声音 + 自己麦克风），同时完全排除浏览器、音乐软件等一切无关声音。

**现有痛点**（本设计必须解决）：
1. 多进程应用（微信/酷狗/Chrome）无法人工判断哪个 PID 在发声，只能盲试；
2. 选错进程时录出无声文件且毫无反馈；
3. loopback 不含麦克风，录不到自己的声音；
4. WAV 文件过大（1 小时 ≈ 600MB）；
5. 命令行操作对日常使用不友好。

## 2. 需求决策记录（头脑风暴结论，均已用户确认）

| # | 决策项 | 结论 |

## 3. 产品设计

### 3.1 主窗口（已按 mockup 确认）

**待机态**：
- "正在发声的软件"列表：每行 = 图标 + 显示名 + exe 名 + 状态（🔊 正在发声 / 安静）；手动"刷新"按钮 + 打开窗口时自动刷新
- 提示文案："没看到想录的软件？让它先出个声，再点刷新"
- "☑ 同时录我的麦克风"复选框，**默认勾选**
- 大按钮"● 开始录音"
- 底部：当前保存目录 + "打开文件夹"链接

**录音态**：
- 大字计时（HH:MM:SS）+ 实时文件大小
- **两条实时电平条**：目标软件声音 / 麦克风（防"空录"的核心反馈）
- 按钮变为"⏹ 停止并保存"

### 3.2 托盘

- 点窗口 ✕ → 缩到托盘继续运行（录音不中断）；双击托盘图标恢复主窗口
- 右键菜单：打开主窗口 / 停止录音 / 退出程序
- 录音中托盘图标叠加红点

### 3.3 设置（⚙）

仅三项：保存文件夹（可改，D8）、输出格式（M4A 默认 / WAV）、麦克风设备（默认系统默认设备，可切换）。配置持久化到本地文件。

### 3.4 意外处理（均已确认）

| 意外 | 应对 |
|------|------|
| 目标进程退出/通话结束未按停止 | 自动停止并完整保存（保留现有进程监控逻辑） |
| 选错进程无声 | 电平条不动，用户当场可见 |
| 磁盘满/写入失败 | 立即提示，尽力保留已录内容 |
| 系统睡眠 | 录音期间 `SetThreadExecutionState` 阻止睡眠（允许关屏） |
| 麦克风被占用/不存在 | 提示并给选择：仅录系统声继续 / 取消 |
| 程序崩溃/断电 | 崩溃保护：输出文件不至于整体报废（见 5.4） |

### 3.5 文件命名

`<保存目录>\<日期> <目标名> <时间>.m4a`，例：`D:\录音\2026-07-17 微信 22-30.m4a`。全局模式用"系统混音"作目标名。

## 4. 架构设计

### 4.1 分层（单进程）

```
┌─ 表现层（阶段二）: Win32 GUI（主窗口/托盘/设置） ─┐
│            或（阶段一）: CLI 参数与控制台输出        │
├─ 引擎层（核心，两阶段共用同一套代码）──────────────┤
│  AudioSessionLister  发声会话枚举（新增）           │
│  CaptureController   录制会话编排（新增）           │
│  CLoopbackCapture    进程/全局环回捕获（现有，复用） │
│  MicCapture          麦克风捕获（新增）             │
│  AudioMixer          双流对齐混音（新增）           │
│  LevelMeter          实时电平计算（新增）           │
│  AudioSink 抽象      WavSink（现有逻辑抽出）        │
│                      M4aSink（新增，MF SinkWriter） │
└─ Windows API: WASAPI / Media Foundation / Win32 ──┘
```

设计原则：引擎层不依赖任何 UI；表现层通过接口 + 回调（电平、状态、错误）与引擎交互。CLI 与 GUI 是同一引擎的两个壳。

### 4.2 关键模块职责

- **AudioSessionLister**：`IMMDeviceEnumerator` → 默认渲染设备 → `IAudioSessionManager2` → `IAudioSessionEnumerator`；每个会话取 `IAudioSessionControl2::GetProcessId()` + `GetState()`（Active=正在发声）；进程名经 `OpenProcess` + `QueryFullProcessImageNameW`；按"同 exe 聚合 + Active 优先"排序输出。
- **MicCapture**：WASAPI 捕获默认输入端点（`eCapture`），事件驱动模式，与现有 loopback 相同的生产者模型，输出统一为 44.1kHz/16bit/立体声（由 `AUDCLNT_STREAMFLAGS_AUTOCONVERTPCM` 负责重采样与声道转换）。
- **AudioMixer**：以 loopback 流为主时钟；麦克风流入独立缓冲，按水位自适应（超限丢样本 / 欠载补静音），瞬时不同步容忍 < 20ms；混合为饱和加法（clamp 至 int16 范围）。
- **LevelMeter**：混音时对两路各算峰值（10–50ms 窗口），经回调上报（CLI 打印字符条 / GUI 画条）。
- **AudioSink**：`write(PCM块)` + `finalize()` 接口。WavSink = 现有 CreateWAVFile/FixWAVHeader 逻辑迁移；M4aSink = `IMFSinkWriter` AAC 编码（44.1kHz/立体声/128kbps）写 MP4 容器。
- **CaptureController**：组装以上模块，管理生命周期（开始/停止/自动停止/错误汇聚），对外暴露简单接口 + 状态回调。现有 `wmain` 的进程监控逻辑并入此层。

### 4.3 阶段一的 CLI 扩展（引擎调试壳）

- `--list`：打印发声会话表（PID / 进程名 / 状态）
- `--mic on|off`：是否混入麦克风（默认 off，保持向后兼容）
- `--format m4a|wav`：输出格式（默认按扩展名推断，冲突时以 --format 为准）
- 录制中控制台加双电平字符条显示
- 现有 `--pid/--mode/--path` 语义不变（三种模式全部保留）

### 4.4 与现有代码的关系

- `CLoopbackCapture` 拆分：捕获逻辑保留；文件写入（WAV 头/WriterThreadProc）移入 WavSink/CaptureController。写入线程模型（队列 + 条件变量）保留并泛化为"消费队列 → AudioSink"。
- 重构顺带修复 Wiki 已知问题：K2（死代码）、K3（Initialize 参数不一致）、K4（队列竞态）、K5（停止状态机死代码）、K7（停止返回值未检查）。K6（4GB 上限）随 M4A 成为默认自然缓解，WAV 路径加上限提示。K1（CMake 编译 .rc + /utf-8）在阶段二打包时处理。
- 遵循用户规范：不删除已验证模块，重构以迁移复用为主。

## 5. 关键技术方案

### 5.1 发声进程列表

音频会话 API 上报的 PID 即"直接提交音频流的进程"，用它作 include 目标必然命中，从根上解决多进程应用盲试问题。列表把同一 exe 的多个会话聚合显示为一项（内部记真实 PID）。

### 5.2 双流同步

两路 WASAPI 流各自事件驱动独立采集，进入各自环形缓冲；混音线程按 10ms 帧节拍消费。时钟漂移（两设备晶振差异）以水位控制吸收。第一版不做重采样对齐（自用场景 + 通话时长 < 数小时，水位法足够）。

### 5.3 AAC 编码

`IMFSinkWriter`（Media Foundation 内置 AAC 编码器，Windows 10 自带，无第三方依赖）。输入 PCM 44.1kHz/16bit/立体声，输出 128kbps AAC / MP4 容器（.m4a）。30 分钟 ≈ 28MB，符合验收标准 4。

### 5.4 崩溃保护

实施时优先验证 MF fragmented MP4（`MFTranscodeContainerType_FMPEG4`）：分段写入使意外中断的文件仍可播放到中断点。若验证不可行或代价过高，降级方案：WAV 临时文件 + 停止时转码 M4A（磁盘换安全）。二选一在 PLAN 阶段的技术验证任务中定案。

### 5.5 防睡眠

录音开始 `SetThreadExecutionState(ES_CONTINUOUS | ES_SYSTEM_REQUIRED)`，停止时恢复。不加 `ES_DISPLAY_REQUIRED`（允许关屏）。

## 7. 验收标准

1. **进程隔离**：微信通话 + 酷狗放歌 + 浏览器视频同时进行，录出文件仅含通话双方声音；
2. **混音完整**：文件中同时清楚听到对方（loopback）与自己（麦克风）；
3. **自动收尾**：录音中退出微信，文件自动完整保存且可正常播放；
4. **防空录**：选择安静进程时电平条静止可辨；
5. **体积**：30 分钟通话输出 ≈ 30MB（M4A 默认档）；
6. **轻快**：单 exe；启动 < 2 秒；录音时 CPU 占用无感；
7. **托盘**：缩到托盘后录音不断，可从托盘停止/恢复窗口。

阶段一结束时 1–5 须在 CLI 下全部达成；阶段二结束时 6–7 加上 GUI 操作路径全部达成。

## 8. 风险与开放问题

| 风险 | 应对 |
|------|------|
| fragmented MP4 在 SinkWriter 的支持细节未验证 | PLAN 中列为先行技术验证任务，有降级方案（5.4） |
| 麦克风与 loopback 时钟漂移长录音下累积 | 水位法 + 阈值丢补；自用短时通话风险低；若实测超预期再引入重采样 |
| 微信通话使用的音频会话可能在通话开始才创建 | 产品层已含应对（先通话后录音的引导文案 + 刷新按钮） |
| 进程环回模式下电平为零的判定（区分"安静"与"没录上"） | 电平条如实显示；文档与 UI 文案不做"没录上"的自动判定，由用户观察 |
